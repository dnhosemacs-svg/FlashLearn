## 1) Agile: qué es y cuál es su objetivo

### Definición

**Las metodologías ágiles como su nombre indica son una serie de metodologías que tienen por objetivo entregar software profesional de calidad en el menor tiempo posible, existen varias metodologías con el mismo fin**

- **Idea principal**:  
  Organizar el trabajo de forma que se realicen todas las tareas de la forma más eficiente posible, obteniendo la mayor calidad en el producto invirtiendo el menor tiempo posible.
- **Cómo cambia la forma de trabajar**:  
  Al usar estas metodologías se mejora tanto la productividad como la motivación del equipo al tener un plan detallado que se va cumpliendo paso a paso.

### Objetivo de Agile en el desarrollo de software

**Entregar una aplicación funcional, trabajando de la forma más eficiente posible.**

- **Qué problema intenta resolver**:  
  Mediante agile se intenta evitar el retraso en las entregas propuestas al cliente, también minimiza la posibilidad de presentar el trabajo con errores y se pretende mejorar la comunicación tanto dentro del equipo como con el cliente para evitar mal entendidos.

- **Qué busca maximizar**:  
  Se pretende mejorar la calidad del software que se entrega a causa de mejorar el feedback tanto interno como externo ya que esta mejora presenta una mejor adaptabilidad en el equipo para resolver los retos que aporte el proyecto.

### Palabras clave

- **Iteración**: Es un ciclo de trabajo en el que se planifica, desarrolla y revisa algo para aprender y mejorar en cada ciclo.  
- **Incremento**: Es una parte terminada del producto que añade valor, ya sea una mejora o nueva funcionalidad.  
- **Entrega continua de valor**: Consiste en entregar mejoras útiles, en lugar de esperar a tener todo el proyecto, para que el cliente vea los avances.  
- **Feedback temprano**: recibir opiniones y detectar fallos cuanto antes mediante una buena comunicación tanto dentro del equipo como con el cliente para corregir los fallos a menos coste.  
- **Adaptación al cambio**: capacidad de ajustar tanto las prioridades como el plan cuando cambien necesidades.

## 2) Scrum: qué es y conceptos principales

### Definición:

**Scrum es un marco de trabajo que se puede aplicar a diferentes proyectos.**

### Conceptos principales

#### Roles

Completa cada rol con 3–5 líneas:

- **Product Owner (PO)**:  
  - Es la persona que actúa del lado del cliente, informando de las necesidades del mismo.
  - Esta figura se coordina con el SM y le presenta un product backlog donde expresa el deseo del cliente.
- **Scrum Master (SM)**:  
  - Es el intermediario que lidia entre el PO y el equipo de desarrollo. 
  - Se reúne con el PO para acordar las exigencias del proyecto y por otro lado debate con el equipo de desarrollo las limitaciones técnicas que presente dicho proyecto y en caso necesario discutirá con el PO posibles cambios de rumbo en caso de ser necesarios.
- **Development Team / Developers (Equipo de desarrollo)**:  
  - Es el equipo de desarrollo propiamente dicho.
  - Se encargará de desarrollar el proyecto, cumpliendo los sprints pertinentes mediante el sprint backlog.

#### Sprints

- **Qué es un sprint**: Es una unidad de tiempo en la que se debe cumplir una serie de objetivos dentro del proyecto.
- **Duración típica**: Se puede organizar en semanas (de lunes a viernes) o en el periodo deseado.
- **Qué se intenta lograr en un sprint**: en cada periodo se debe hacer una planeación, priorizando tareas más importantes, luego se tendrá un tiempo de desarrollo y al finalizar el periodo se tendrá una review con su retrospectiva para ir mejorando.

#### Backlog

Define y diferencia:

- **Product Backlog**: Es una lista de tareas hecha por el PO en la que añade todas las tareas necesarias para la realización del proyecto, ordenadas por prioridad. 
- **Sprint Backlog**: A raiz del product backlog, el equipo de desarrollo realiza otra lista que incluye todas las tareas que se pretenden realizar en el sprint actual, dicha lista estará ordenada también por prioridad, respetando el orden del product backlog. 

#### Reviews :

- Una Sprint review es una reunión informal donde se reúnen el equipo Scrum y los stakeholders.

Explica:

- **Qué ocurre en una Sprint Review**: Dicha reunión se celebra con una estructura, primero el PO hace una presentación para recordar el objetivo del sprint, luego el equipo de desarrollo expone el incremento logrado en dicho sprint mediante demostración de funcionalidad y por ultimo se discuten los siguientes pasos a seguir y en caso de que sea necesario se modifica el product Backlog. 
- **Qué salida deja**: Una vez terminada la reunión se tendrá un feedback completo del estado del proyecto, así como una conclusión del incremento conseguido que facilitará la toma de decisiones futuras.

## 3) Kanban: qué es y cómo se usa para organizar tareas

### Definición 

**Kanban es una metodología que, mediante una presentación visual ayuda a gestionar un proyecto.**
**Mediante Kanban se pretende encontrar posibles cuellos de botella y solucionarlos para que el trabajo se mantenga en un rendimiento óptimo.**

### Cómo se usa para organizar tareas 

Describe un tablero Kanban típico:

- **Columnas típicas**: En el tablero que usaremos se añadirán las columnas (por hacer, en proceso y hecho) y dentro de cada unas estará una tarjeta por cada tarea que esté dentro de la columna.
- **Qué representa una tarjeta**: Cada tarjeta es una tarea de nuestro proyecto.
- **Cómo se mueve el trabajo**: Dentro de nuestro panel iremos moviendo cada tarea de una columna a otra según se avance en el trabajo.

### Conceptos clave

- **WIP limit (límite de trabajo en progreso)**: Esto es el límite de tareas que un equipo puede afrontar a la vez, es importante limitar las tareas para no saturar al equipo de trabajo.  
- **Flujo (flow)**: Es el recorrido que sigue una tarea desde que se crea hasta que se completa.   
- **Bottleneck / cuello de botella**: Se origina cuando una tarea impide que se puedan completar otras tareas diferentes, ralentizando o hasta deteniendo el flujo de trabajo.

## 4) Diferencias entre Scrum y Kanban

Completa esta comparación con tus palabras. Intenta dar ejemplos cortos.

- **Cadencia de trabajo**:  
  - Scrum: Se trabaja en sprints, por lo que la entrega se hace en ciclos cerrados.  
  - Kanban: Se trabaja con flujo, moviendo tareas según avanza el equipo.
- **Planificación**:  
  - Scrum: La planificación se hace al inicio de cada sprint.  
  - Kanban: La planificación se ajusta en función del estado del tablero.
- **Cambios de prioridad**:  
  - Scrum: Durante el sprint no se cambia la prioridad para no afectar al objetivo.  
  - Kanban: Se pueden cambiar prioridades en cualquier momento, reordenando tareas según urgencia.
- **Roles definidos**:  
  - Scrum: Tiene roles claramente definidos, con responsabilidades concretas.  
  - Kanban: No exige roles y tiene una estructura más libre.
- **Medición y mejora**:  
  - Scrum: Se mejora al final de cada sprint, analizando el incremento y el trabajo del equipo de desarrollo.  
  - Kanban: Se mejora observando el flujo de trabajo y detectando cuellos de botella para optimizar el rendimiento.

## 5) ¿En qué situaciones usar cada metodología?

### Cuándo usar Scrum

- **En mi opinión Scrum viene muy bien cuando tenemos un equipo compuesto por varias personas, el cual debe hacer un proyecto grande, con multitud de tareas que requiera bastante tiempo de desarrollo.**

### Cuándo usar Kanban

- **Por una parte veo interesante usar kanban junto a scrum, ya que una vez organizado un sprint podemos organizar dichio sprint mediante kanban para optimizar el flujo de trabajo**

- **Por otro lado se puede usar kanban independientemente de scrum, en proyectos donde se tenga menos número de tareas o cuando el equipo es relativamente reducido, ya que en caso de no ser así pienso que el tablero sería demasiado grande, perdiendo su utilidad original y volviendose tedioso de trabajar.**
